The Filesystem Is Not a Drive

the assumption you carry

On Windows, your world starts with drives: C:\, D:\, E:\. Each drive is its own island. Storage is the filesystem. The physical device and the logical location are mentally the same thing.

Windows: "My files are on the C drive." — The drive is the location.

Linux: "My files are at /home/user/." — The location is a point in a single tree. The drive underneath is irrelevant to your path.

the linux reality

Linux has one tree. It starts at / (root). Everything — USB drives, network shares, other physical disks — gets mounted into this single tree. A USB might appear at /media/usb0. Another disk might be mounted at /data. No letters. No islands.

/ # ← the entire universe starts here
├── home/ # user files (Windows: C:\Users\)
│ └── yourname/
│ ├── Documents/
│ └── Downloads/
├── etc/ # system config files
├── usr/ # programs (Windows: Program Files)
├── bin/ # core executables
├── dev/ # devices ARE files here
└── media/ # USB drives, external disks mount here
Key insight: When you plug in a USB drive on Linux, the OS doesn't assign it a letter — it attaches it to the tree at a mount point. The folder /media/usb0 suddenly has contents. Unplug it, the folder is empty again. The tree is always the tree.
Q1. On Linux, you have a second hard drive. Which of these best describes where its files appear?
Correct. The drive is mounted at a directory. From that point, it's just a folder in the tree. The physical reality is abstracted away — this is intentional. Linux doesn't care what the hardware is; it only cares about the tree.

Paths: Forward Slashes & Roots

⊞ Windows Paths
  • C:\Users\John\Documents\file.txt
  • Backslash separators
  • Drive letter prefix required
  • Case-insensitive (File.txt = file.txt)
  • Spaces common in paths
  • Relative: .\subfolder\
◌ Linux Paths
  • /home/john/documents/file.txt
  • Forward slash separators
  • Always starts from / (or ~)
  • Case-sensitive (File.txt ≠ file.txt)
  • Spaces are pain — avoid them
  • Relative: ./subfolder/
special path shorthands

~ — your home directory (/home/yourname)
. — current directory
.. — parent directory
/ — root of the entire system (nothing above this)

path examples cd /home/user/projects # absolute path — always works cd ~/projects # ~ expands to /home/user — same result cd ./subfolder # relative — from wherever you currently are cd .. # go up one level cd ../../etc # up two, then into etc/

Case sensitivity will bite you. On Windows, MyFile.txt, myfile.txt, and MYFILE.TXT are the same file. On Linux they are three completely different files that can coexist in the same directory. Many beginners spend an hour debugging what is simply a capital letter.

Q2. You're currently at /home/user/projects/webapp. What does cd ../../documents land you?
Correct. ../ goes to /home/user/projects, then ../ again goes to /home/user, then documents descends into it. Each .. climbs exactly one level.
path-practice — simulated terminal

You are at: /home/user/projects/webapp/src

Type the command to reach /home/user/projects using a relative path:

user@linux:~/projects/webapp/src$

Everything Is a File

the deepest linux principle

Linux treats nearly everything as a file. Your keyboard: a file in /dev/input/. Your hard disk: /dev/sda. A network socket: a file descriptor. System settings: text files in /etc/. This isn't just trivia — it means the same tools (read, write, pipe) work on almost everything.

Windows: Settings live in the Registry (a binary database). Device properties live in Device Manager. Each type of thing has its own special tool to access it.

Linux: Want to change a system setting? Open a file in /etc/ with a text editor. Want to read from a hardware device? Open its file in /dev/. The tool is always the same.

/dev/ # hardware as files
├── sda # your primary hard drive
├── sda1 # partition 1 of that drive
├── tty0 # your terminal
└── null # black hole — write anything, it vanishes
/etc/ # system config as text files
├── hostname # one line: your machine's name
├── hosts # like Windows hosts file, but it matters more
└── passwd # user accounts (no binary registry)
Practical consequence: On Windows you'd open Device Manager or Registry Editor with a GUI to change system behavior. On Linux you often just nano /etc/some-config-file and save. The learning curve is steep at first, but the surface area of knowledge stays small — it's always files, always text.
Q3. You want to change your machine's hostname on Linux. What's the most Linux-native approach?
Exactly. /etc/hostname contains a single line with your machine's name. Open it, change it, save. That's it. This is the "everything is a file" principle in its most direct form.

Permissions & Ownership

where windows misleads you

Windows has permissions (ACLs), but they're largely invisible to most users. You right-click → Properties → Security and there they are — but most people never touch them. Linux makes permissions unavoidable, visible, and foundational to everything.

reading permissions ls -l -rw-r--r-- 1 user group 1234 Feb 14 file.txt drwxr-xr-x 2 user group 4096 Feb 14 myfolder/ -rwxr-xr-x 1 root root 8192 Feb 14 script.sh
decoding the permission string

-rw-r--r-- breaks down as:

- = type (- = file, d = directory)
rw- = owner can read + write
r-- = group can only read
r-- = everyone else can only read

r = read   w = write   x = execute   - = denied

changing permissions # Make a script executable (you MUST do this before running it) chmod +x script.sh # Give owner read+write+execute, group read, others nothing chmod 750 script.sh # Change owner of a file (requires sudo) sudo chown username:groupname file.txt

The executable bit is invisible on Windows. On Windows, if a file ends in .exe or .bat, it runs. On Linux, the extension means nothing. A file named script.sh does NOT run until you explicitly set the execute bit with chmod +x. This is security by default, not convenience by default.

Q4. You download a shell script and try to run it with ./setup.sh — you get "Permission denied". What's the most likely fix?
Correct. chmod +x setup.sh sets the execute bit. The error "Permission denied" when you own the file almost always means the execute bit is missing, not that you lack ownership. sudo would not help here — root also can't execute a file without the execute bit on most Linux systems.

The Terminal Is Not CMD

⊞ Windows CMD / PowerShell
  • Optional — most tasks have a GUI
  • Limited composability
  • Output is objects (PowerShell)
  • Commands: dir, copy, del
  • Rarely used for system config
  • Limited scripting power
◌ Linux Bash / Zsh
  • Primary interface — GUI is optional
  • Everything pipes together
  • Output is text — universally
  • Commands: ls, cp, rm
  • Terminal IS system configuration
  • Scripting is first-class
essential commands (windows → linux mapping) # Windows: dir → Linux: ls -la # list all files with details # Windows: cd → Linux: same cd ~/projects # Windows: copy → Linux: cp file.txt backup.txt # Windows: move → Linux: mv file.txt ./other/ # also used to rename files # Windows: del → Linux: rm file.txt rm -rf folder/ # remove folder recursively (no recycle bin) # Windows: mkdir → Linux: same mkdir -p a/b/c # -p creates parent dirs too

rm -rf has no undo. There is no Recycle Bin. No Ctrl+Z. When you delete in the terminal, it's gone. This is not a bug — it's intentional Unix philosophy. This is also why you'll see the warning: never run sudo rm -rf / — that's deleting the entire system.

Pipes are the superpower. The | character connects commands: the output of one becomes the input of the next. ls -la | grep ".txt" lists files, then filters to only show .txt ones. This composability is what makes the terminal more powerful than any GUI.
pipe examples ls -la | grep ".sh" # find all .sh files cat log.txt | grep "ERROR" # find error lines in a log ps aux | grep "python" # find running python processes history | grep "git" # find all git commands you've run
Q5. You want to see a list of all running processes, but only ones that contain the word "node". What command chain does this?
Correct. ps aux dumps all processes as text. That text is piped into grep "node", which filters to only lines containing "node". This is the Unix pipe at its purest: two dumb tools producing something smart together.

Cloning: What It Actually Does

what you probably thought

If you've used Windows and maybe GitHub Desktop or downloaded a zip: "cloning" felt like downloading a folder. You get the files. You can look at them. Maybe run them if you're lucky.

what git clone actually does

Git clone downloads the entire history of a project, not just the current files. Every commit, every branch, every contributor's changes — compressed into a hidden .git/ folder. What you see in the directory is just the checked out version of the latest commit.

cloning a repository # Basic clone — downloads repo into a folder with the repo's name git clone https://github.com/user/repo.git # Clone into a specific folder name git clone https://github.com/user/repo.git my-project # What you now have: my-project/ ├── .git/ ← entire history lives here (don't touch) ├── README.md ├── src/ └── package.json # Verify it worked — see the latest commits cd my-project git log --oneline

Think of .git/ as the project's DNA. The files you see are the organism. The .git/ folder is the genome — containing every mutation the project ever went through. You can travel back to any point in history, create branches, merge changes. Never delete .git/. If you do, you have files but no history, no version control.

Clone vs Download ZIP: GitHub lets you download a ZIP. That gives you the files but no .git/ folder. You cannot commit, push, or track changes. Always clone when you intend to work on the code.
after cloning — checking status git status On branch main Your branch is up to date with 'origin/main'. nothing to commit, working tree clean git remote -v origin https://github.com/user/repo.git (fetch) origin https://github.com/user/repo.git (push) # "origin" is the name for the remote you cloned from # You can have multiple remotes (e.g., upstream for the original project)
Q6. You clone a repo and accidentally delete the .git/ folder. What have you lost?
Correct. The .git/ folder IS the repository. Without it, you have plain files with no history, no branches, no remote connection. You'd need to re-clone. The files themselves are untouched, but they're no longer a git repository.

Building From Source

the windows expectation

On Windows, you rarely build software. You run an installer (setup.exe), click Next several times, and the program appears in your Start Menu. Someone else compiled everything. You consume the result.

the linux difference

Linux gives you the source code and the tools to compile it yourself. This is power — you can modify anything, understand what you're installing, and build for your exact system. It's also responsibility — you need to understand the process.

the typical build-from-source flow # Step 1: Clone or download the source git clone https://github.com/project/tool.git cd tool # Step 2: Install dependencies (what the project NEEDS to build) sudo apt install build-essential libssl-dev # Step 3: Configure (checks your system, sets build options) ./configure # Step 4: Compile (converts source code → binary executable) make # Step 5: Install (copies the binary to /usr/local/bin so it runs anywhere) sudo make install

make is not magic — it's a recipe. The Makefile in the project directory is literally a list of instructions: "to build this, run these compiler commands in this order." When you run make, you're executing that recipe. When it fails, the error tells you exactly which ingredient (dependency) is missing.

Modern projects often skip the classic ./configure && make pattern. You might see cmake, meson, cargo build (Rust), or go build instead. The concept is the same: configure, compile, install. The tools change, the flow doesn't.
when build fails — typical dependency error checking for ssl/ssl.h... no configure: error: OpenSSL not found. Install libssl-dev and try again. # The fix: install what's missing sudo apt install libssl-dev # Then re-run configure ./configure
Q7. You run ./configure and it fails saying a library is not found. What's the correct next step?
Correct. The configure script checks your system for all the libraries the project needs. If one is missing, you install it (usually sudo apt install lib-something-dev — notice the -dev suffix, which includes the header files needed for compilation), then re-run ./configure. Permissions won't fix a missing library.

Package Managers vs Installers

⊞ Windows: Manual Installation
  • Google the software name
  • Find download page
  • Download .exe or .msi
  • Run installer wizard
  • Updates? Google it again
  • Uninstall? Add/Remove Programs
◌ Linux: Package Manager
  • sudo apt install git
  • That's it. Done.
  • Updates: sudo apt upgrade
  • Remove: sudo apt remove git
  • Verified, secure packages
  • Dependency resolution automatic
apt — the package manager for debian/ubuntu # Update the list of available packages (do this first) sudo apt update # Upgrade all installed packages to latest versions sudo apt upgrade # Install a package sudo apt install nodejs # Search for a package apt search image editor # See info about a package before installing apt show nodejs # Remove a package sudo apt remove nodejs # Remove package AND its config files sudo apt purge nodejs

apt is the App Store, but for the entire OS. Everything — system tools, development libraries, GUI apps, fonts — comes from the same vetted source. On Windows you'd visit 12 different websites to set up a dev environment. On Linux: one terminal, one command per tool. The package manager also tracks every dependency, so uninstalling one thing cleanly removes everything it depended on.

Different distros, different managers: Ubuntu/Debian → apt. Fedora/RHEL → dnf. Arch → pacman. The commands differ, but the concept is identical. If you learn apt, understanding the others is minutes of reading, not hours of relearning.
Q8. You want to install git on Ubuntu. sudo apt install git returns "Unable to locate package git". What's the most likely fix?
Correct. apt update downloads the latest list of available packages from the repository servers. Without a recent update, apt searches an outdated local cache and may not know the package exists. Always sudo apt update before installing anything, especially on a fresh system.
FINAL. A friend sends you a GitHub link to a Node.js project and says "just build it and run it". Put these steps in your head: what's the complete sequence on a fresh Linux install?
Correct. The sequence is: install the runtime tools (git, node, npm) via apt → clone the repo (gets source + .git history) → cd into it → npm install (installs JS dependencies from package.json) → npm start (runs it). This complete flow — tooling, clone, dependencies, run — is the Linux dev loop.
